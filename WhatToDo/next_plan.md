# Smart City UI & UX Enhancements Plan

This document outlines the architecture and implementation steps for the next phase of the Smart City project, focusing on advanced data visualization, service workflow state management, and improved resident user experience.

## 1. Rule-Specific Visualizations (Analyst/Manager Dashboard)

To effectively communicate the insights generated by the 10 analysis rules, we will implement the following specific visual representations in the Flutter UI.

### Energy Domain
1. **Power Balance (`power_balance`)**: 
   - *Plot*: **Overlaid Area Chart**.
   - *Details*: Green area for Solar Generation, Red/Blue area for Campus Consumption. Instantly highlights periods of deficit.
2. **Solar Efficiency (`solar_efficiency`)**: 
   - *Plot*: **Gauge Chart** & **Trend Line**.
   - *Details*: A real-time gauge showing current efficiency % against expected theoretical output, accompanied by a 24h trend line.
3. **Consumption Anomaly (`consumption_anomaly`)**: 
   - *Plot*: **Line Chart with Confidence Bands (Bollinger Bands)**.
   - *Details*: Shows the historical moving average with shaded bands. Anomalous spikes breaching the upper band are marked with red scatter dots.
4. **Grid Stability (`grid_stability`)**: 
   - *Plot*: **Dual-Axis Line Chart**.
   - *Details*: Left Y-axis for Voltage (V), Right Y-axis for Frequency (Hz). A shaded safe zone allows visual confirmation of stability.
5. **Device Schedule (`device_schedule`)**: 
   - *Plot*: **Gantt Chart / Heatmap Timeline**.
   - *Details*: Rows represent devices (e.g., Streetlights). X-axis is time. Colors indicate actual state vs. expected scheduled state.

### EHS Domain
1. **Air Quality (`air_quality`)**: 
   - *Plot*: **Radar (Spider) Chart**.
   - *Details*: Axes for PM2.5, CO2, NO2, and O3. A polygon overlay represents the safe limits. If the data polygon spikes outside the safe polygon, it's visually obvious.
2. **Indoor Comfort (`indoor_comfort`)**: 
   - *Plot*: **Scatter Plot over ASHRAE Envelope**.
   - *Details*: X-axis: Temperature, Y-axis: Humidity. A static green polygon represents the ideal comfort zone. Current readings are plotted as dots.
3. **Water Safety (`water_safety`)**: 
   - *Plot*: **Divergence Line Chart**.
   - *Details*: Plots Pressure and Flow Rate. A calculated "divergence" area highlights when high flow meets low pressure (pipe burst signature).
4. **Water Quality (`water_quality`)**: 
   - *Plot*: **Multi-line Chart with Threshold Lines**.
   - *Details*: Tracks pH, Turbidity, and Chlorine. Horizontal dashed lines represent the safe upper and lower bounds for each metric.
5. **Equipment Health (`equipment_health`)**: 
   - *Plot*: **Step Chart (ON/OFF) overlaid with Line Chart (Flow)**.
   - *Details*: Visually demonstrates "Dry Running" by showing a high step (Pump ON) while the overlaid line (Flow Rate) remains at zero.

## 2. Interactive Node Telemetry UI
*Pattern Applied: **Strategy Pattern** (for dynamic metric selection).*

**Goal:** Allow users to filter which parameters are plotted and view all values simultaneously on hover.
- **Implementation**:
  - Update `NodeDetailScreen` to fetch the complete JSON payload keys.
  - Render a row of selectable `FilterChip` widgets (e.g., `[x] Voltage [ ] Current [x] Power`).
  - Use the **Strategy Pattern** in the chart builder: Depending on the active chips, the chart data strategy dynamically generates the `LineChartBarData` arrays.
  - Implement `LineTouchData` in `fl_chart` to render a custom tooltip. When the user taps/hovers on an X-axis point (timestamp), the tooltip loops through all parameters and displays their exact values at that specific time.

## 3. Servicer/Technician Workflow & History
*Pattern Applied: **State Pattern**.*

**Goal:** Allow technicians to complete assignments and view their past service history.
- **Backend Changes**:
  - Add a `status` Enum to `ServicerAssignment` (`ASSIGNED`, `IN_PROGRESS`, `RESOLVED`).
  - Add a `resolution_notes` text field and `resolved_at` timestamp.
  - Create endpoint: `PUT /servicer/assignments/{id}/state` utilizing the State Pattern to handle valid transitions (e.g., cannot move from `RESOLVED` back to `ASSIGNED` without a manager).
- **Frontend Changes**:
  - Update `ServicerDashboard` to utilize a `TabBar`: "Active Tasks" and "Service History".
  - Add a "Mark Complete" action button to active tasks. This opens a dialog prompting for mandatory `resolution_notes` before firing the PUT request.
  - The "History" tab will fetch and display assignments where `status == RESOLVED`.

## 4. Resident Zone Discovery
**Goal:** Prevent a blank screen when residents open the subscription panel.
- **Implementation**:
  - Update `SubscriptionPanel` (or `ResidentDashboard` search area).
  - On widget initialization (`initState`), if the search query is empty, immediately fetch and display the root catalog of zones using `zoneNodesProvider`.
  - The search bar will act as a local filter over this fully populated list, providing immediate context to the user about what zones are available to subscribe to.

# IEEE 42010 Architecture Framework

This document represents the formal architectural description for the Smart City Living Lab, mapped according to the IEEE 42010 standard. It acts as a single cohesive artifact establishing clear traceability from stakeholders through to concrete system views.

## 1. Stakeholders

| ID | Stakeholder | Description |
| :--- | :--- | :--- |
| **SH1** | Energy Management Team | Focuses on optimizing campus solar generation and AC load balancing. |
| **SH2** | EHS Management Team | Monitors environmental vitals like air (AQI) and water quality. |
| **SH3** | Crowd & Access Management (CAM) | Secures campus entrances and monitors real-time crowd density. |
| **SH4** | Researchers (Data Scientists) | Requires historical data access to develop and test ML models. |
| **SH5** | Residents (End-Users) | Beneficiaries relying on the system for safety alerts and automation. |

---

## 2. Architectural Concerns

Concerns represent the primary interests generated by stakeholders that drive the system design.

| ID | Concern Name | Description |
| :--- | :--- | :--- |
| **C1** | Sub-second Latency | Entrance authentication processing must complete in <1s. |
| **C2** | PII Anonymization | The system must strictly anonymize or eliminate Personal Identifiable Information. |
| **C3** | Hardware Interoperability | The system must seamlessly integrate 300 highly heterogeneous protocols/sensors. |
| **C4** | Edge Node Availability | Outdoor sensors must conserve battery while maintaining reliability. |
| **C5** | Historical Telemetry Retrieval | Time-bounded (30-day) sensor data must be safely exposed for analysis. |
| **C6** | Real-time Situation Awareness | Dashboards and alerts must reflect near real-time city parameters. |

---

## 3. Viewpoints

Viewpoints serve as reusable architectural abstractions defining *how* specific classes of concerns are addressed across the system. 

| ID | Viewpoint | Addresses Concerns |
| :--- | :--- | :--- |
| **VP1** | **Performance & Scale** | C1, C3, C4 |
| **VP2** | **Security & Privacy** | C2 |
| **VP3** | **Information Architecture** | C5 |
| **VP4** | **Functional Interaction** | C6 |

---

## 4. Views

Views operate under governed viewpoints to provide concrete system representations resulting from our technical decisions.

| ID | View | Governed By | Description |
| :--- | :--- | :--- | :--- |
| **V1** | **Edge-Processing Timings** | VP1 | Process diagrams outlining the sub-second local authentication pathways. |
| **V2** | **Data Scrubbing Pipeline** | VP2 | Visualizes the decoupled middleware engine stripping PII before storage. |
| **V3** | **Access Control Matrix** | VP2 | The tabular RBAC mapping linking stakeholders to their permitted actions. |
| **V4** | **Researcher API Gateway** | VP3 | Swagger/OpenAPI specifications exposing the 30-day Time-Series Database bounds. |
| **V5** | **Live Telemetry Dashboards** | VP4 | The Flutter-rendered management web interfaces representing sensor metrics. |
| **V6** | **Hardware Serviceability Log** | VP1, VP4 | The maintenance heartbeat representation tracking battery and calibration status. |

---
